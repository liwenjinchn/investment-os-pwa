# Investment OS：事件决策 v1

## 交付结论

事件决策已成为 PWA 的正式主流程。它解决的不是“AI 看完新闻告诉我买什么”，而是一个更严格的问题：这条事件是否有足够证据改变原始论点，情景与下行是否闭合，组合纪律是否允许动作，以及如何在事后判断当时的过程是否正确。

## 五段闭环

### 1. 定义事件

- 关联标的或公司
- 事件类型、已知日期或时间窗
- 只含已确认内容的事实摘要
- 被影响的原始投资论点
- 可观察的论点证伪条件

### 2. 建立证据账本

每条证据都保存标题、日期、来源链接、证明内容，以及两组独立标签：

- 证据等级：一手文件、公司口径、独立来源、市场信号、未核实
- 内容性质：事实、假设、判断

进入决策至少需要两条有效证据，其中至少一条是“一手文件”或“公司口径”的事实。未核实内容若被标成事实会触发警告。

### 3. 构建情景树

每个情景包含概率、组合影响百分点、论点影响、概率依据和下一可观察信号。

- 概率必须合计 100%，否则不计算期望影响。
- 必须存在至少一个组合影响为负的情景。
- 每个情景都必须有依据与信号。
- 组合期望影响 = Σ（情景概率 × 组合影响）。

这不是目标价模型；它是把路径、下行和组合后果写清楚的纪律工具。

### 4. 纪律决策

系统只生成可解释的流程建议：不动、复盘、减仓、加仓、写入观察。用户仍需确认最终动作并写明理由。

以下情况会阻止确认：

- 事件、标的、时间、论点或证伪条件缺失
- 一手/公司事实和交叉证据不足
- 情景概率不等于 100%
- 没有负面情景
- 情景缺少依据或可观察信号

现金下限、单只仓位上限和主题暴露上限可否决加仓。即使事件期望影响偏正面，也不能用事件理由绕开组合规则。

### 5. 监控与复盘

确认时必须写明下一触发器和复盘日期。到期后追加：

- 实际结果
- 可复用教训

完成复盘后事件归档，但原事件、证据、概率、动作和理由不会被覆盖。

## 数据与隐私

- 所有持仓、事件、证据、情景、规则、日志和 API 设置都保存在当前浏览器的 IndexedDB。
- 数据库从 `investment-os-local` v1 自动升级到 v2，旧持仓、规则、日志和复盘数据不需要迁出。
- 备份格式升级到 v2，并兼容缺少 `eventCases` 的旧备份。
- 事件决策的确定性规则完全在本地运行。
- 只有用户主动点击“让 AI 只做反方审查”时，事件摘要、证据与情景才会发送到用户配置的 AI Base URL。
- API Key 仍只保存在本地浏览器；新导出的 backup.json 会主动清空 API Key，避免备份文件泄露凭证。恢复这种备份时，当前设备已有的 Key 不会被空值覆盖。

## 已知边界

- v1 不自动抓取新闻、公告或价格；来源由用户录入，因此不会把未经选择的信息流变成交易信号。
- v1 的组合影响由用户以百分点填写，不替代估值模型、实时市场数据或交易执行系统。
- “加仓 / 减仓”是纪律工作流标签，不是自动交易指令，也不是收益承诺。
- 如果需要真实证券的事件研究，关键事实、市场价格与日期仍应以当时的一手文件和市场数据为准。
